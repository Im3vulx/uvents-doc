# Historique

## Objectif de cette section

Cette page présente un **historique synthétique** du projet **ONY**.

L’objectif est de conserver une trace lisible des grandes évolutions du projet, en particulier lorsque certains éléments de la documentation, de l’infrastructure ou du code conservent des héritages plus anciens.

Cette page n’a pas vocation à remplacer l’historique Git détaillé, mais à expliciter les transformations structurantes utiles à la compréhension globale du projet.

## Origine du projet

Le projet aujourd’hui nommé **ONY** a d’abord été porté sous le nom **Uvents**.

Cet héritage reste encore visible dans plusieurs éléments techniques, notamment :

- certains noms de dépôt ;
- certaines URLs ;
- certains scripts ;
- certaines conventions de chemin ;
- certaines références historiques dans la documentation ou l’infrastructure.

## Changement de nom

Le projet déployé s’appelle désormais **ONY**.

Ce changement de nom constitue un point important de lecture documentaire, car il explique pourquoi certains éléments du projet ne sont pas encore totalement harmonisés sur le plan du nommage.

Il faut donc distinguer :

- le nom actuel de référence : **ONY** ;
- les références historiques encore présentes : **Uvents**.

## Conséquences techniques de l’historique

Le changement de nom n’implique pas que tous les éléments techniques aient déjà été renommés.

Dans l’état actuel, il est normal de retrouver encore des traces de l’ancien nom dans :

Cela ne constitue pas nécessairement une erreur, mais un héritage technique à connaître pour bien lire le projet.

 
## Intérêt de conserver cet historique

Documenter cet historique est utile pour plusieurs raisons :

- éviter les incompréhensions lors de la lecture des fichiers ;
- expliquer les écarts de nommage ;
- faciliter l’onboarding sur le projet ;
- rendre la documentation plus honnête et plus claire ;
- garder une mémoire des évolutions structurantes.

## Harmonisation progressive

L’historique ne doit pas empêcher l’évolution.

Au contraire, cette page permet d’assumer clairement que :

- le projet a changé de nom ;
- certains héritages subsistent ;
- une harmonisation progressive peut encore être menée ;
- la documentation doit aider à faire le lien entre passé technique et état actuel.

## Position recommandée dans la documentation

La bonne posture consiste à ne pas masquer l’historique, mais à le présenter clairement.

Cela permet :

- de réduire l’ambiguïté ;
- de mieux comprendre certaines incohérences apparentes ;
- d’éviter de prendre un héritage ancien pour une erreur récente.

## À retenir

Le projet actuel s’appelle **ONY**, mais il conserve encore des traces historiques de son ancien nom **Uvents**.

Cette page sert à expliquer cet héritage et à clarifier la lecture des éléments techniques ou documentaires qui n’ont pas encore été entièrement renommés.

L’historique fait partie de la compréhension du projet, au même titre que son état présent.
