# Objectifs

## Objectif global

L’objectif principal d’ONY est de construire une application web progressive capable de **faciliter la découverte, la consultation, la réservation et la gestion d’événements locaux**, dans une expérience pensée en priorité pour le mobile.

Le projet vise à démontrer qu’il est possible de proposer une solution cohérente, maintenable et évolutive avec un socle technique moderne, tout en gardant une mise en œuvre réaliste dans un contexte de MVP.

## Objectifs produit

### 1. Faciliter la découverte locale

Permettre à l’utilisateur de trouver rapidement des événements autour de lui via :

- une carte interactive ;
- des listes contextuelles ;
- des filtres ;
- une recherche simple ;
- des recommandations cohérentes.

### 2. Rendre la consultation plus fluide

Réduire la friction entre la découverte et l’accès à l’information utile grâce à :

- des résumés d’événements lisibles ;
- une page détail claire ;
- une hiérarchie visuelle forte ;
- un design mobile-first.

### 3. Fournir un socle de billetterie

Proposer un parcours de réservation et de ticketing simulé permettant :

- de réserver un événement ;
- de générer un billet ;
- d’associer un QR code ;
- de scanner et valider un billet à l’entrée.

### 4. Ouvrir un parcours organisateur

Préparer un parcours dédié aux organisateurs pour :

- créer des événements ;
- les modifier ;
- gérer leur présence dans l’application ;
- disposer d’un cadre vérifiable côté données et sécurité.

## Objectifs techniques

### 1. Poser une architecture cohérente

Le projet doit reposer sur une architecture claire, documentée et suffisamment modulaire pour permettre :

- la maintenance ;
- la reprise ;
- l’évolution progressive ;
- l’ajout de nouvelles fonctionnalités sans désorganiser l’existant.

### 2. S’appuyer sur des technologies modernes

Le projet utilise un stack web moderne afin d’assurer :

- rapidité de développement ;
- lisibilité du code ;
- bonne intégration entre front, auth, base et services externes ;
- déploiement simple et supervision exploitable.

### 3. Garantir un niveau correct de sécurité

Les choix techniques doivent permettre :

- une authentification robuste ;
- une séparation claire entre données publiques et sensibles ;
- une bonne gestion des secrets ;
- un encadrement des accès aux données.

### 4. Assurer la capacité d’évolution

Le MVP ne constitue pas un point final.Il doit servir de base pour aller vers :

- des paiements plus complets ;
- une meilleure recommandation ;
- des statistiques plus avancées ;
- une logique organisateur plus aboutie ;
- une meilleure industrialisation des tests.

## Objectifs opérationnels

### 1. Centraliser la documentation

La documentation doit devenir la référence principale du projet, afin d’éviter la dispersion des informations entre :

- fichiers locaux ;
- exports PDF ;
- documentation partielle dans le code ;
- notes informelles.

### 2. Faciliter l’exploitation

Le projet doit pouvoir être :

- déployé ;
- supervisé ;
- diagnostiqué ;
- relancé ;
- rollbacké
  de manière claire et reproductible.

### 3. Structurer la collaboration

Le projet doit rester lisible pour :

- le ou les développeurs actuels ;
- un futur repreneur ;
- un encadrant ;
- un collaborateur ponctuel.

## Objectifs pédagogiques

Le projet sert également de support de montée en compétence et de démonstration de maîtrise sur plusieurs dimensions :

- architecture web full-stack ;
- documentation technique ;
- CI/CD ;
- exploitation ;
- UX/UI ;
- accessibilité ;
- modélisation de données ;
- organisation de projet.

## Objectifs de long terme

À plus long terme, ONY a vocation à dépasser le simple MVP local pour devenir une base réutilisable et extensible.
L’objectif n’est pas seulement d’avoir un prototype fonctionnel, mais de construire un socle suffisamment sérieux pour être prolongé, testé, documenté et amélioré dans la durée.
