# Référentiels

## Objectif de cette section

Cette page présente les principaux **référentiels** utiles pour encadrer l’accessibilité et la qualité dans le projet **ONY**.

L’objectif est de préciser :

- quelles bases de référence peuvent guider le projet ;
- pourquoi elles sont utiles ;
- comment elles s’intègrent dans une démarche réaliste ;
- en quoi elles servent de cadre sans bloquer l’avancement du produit.

## Principe général

L’accessibilité et la qualité ne doivent pas être traitées comme des ajouts tardifs ou purement formels.

Elles reposent sur des principes, des bonnes pratiques et des référentiels qui permettent de structurer les choix de conception, de développement et de validation.

Dans ONY, l’objectif n’est pas de prétendre à une perfection théorique immédiate, mais de s’appuyer sur des repères clairs pour construire une amélioration progressive et crédible.

## Référentiels d’accessibilité

### WCAG

Les **WCAG** constituent l’un des référentiels les plus connus en matière d’accessibilité numérique.

Ils fournissent une base de principes pour rendre les interfaces :

- perceptibles ;
- utilisables ;
- compréhensibles ;
- robustes.

Même sans viser une formalisation exhaustive de tous les critères dès maintenant, ils fournissent un cadre utile pour orienter les choix de conception.

### RGAA

Dans un contexte francophone, le **RGAA** constitue également un référentiel important.

Il permet de traduire les exigences d’accessibilité dans une logique plus opérationnelle et plus proche des vérifications concrètes.

Pour ONY, il représente une base pertinente pour structurer une lecture plus concrète des exigences d’accessibilité, en particulier si le projet cherche à professionnaliser davantage sa démarche qualité.

## Référentiels de qualité

La qualité ne repose pas sur un seul document de référence, mais sur un ensemble de principes de bon sens technique et produit.

Elle concerne notamment :

- la cohérence de l’interface ;
- la clarté des parcours ;
- la lisibilité des contenus ;
- la stabilité du comportement ;
- la prévisibilité des actions ;
- la limitation des erreurs d’usage.

Dans ONY, la qualité doit être comprise comme une exigence transversale, à l’intersection de l’UX, de l’implémentation et des tests.

## Référentiels internes

Au-delà des standards externes, le projet peut aussi s’appuyer sur ses propres repères internes.

Cela peut inclure :

- des conventions UI ;
- des règles de navigation ;
- des habitudes de structuration de composants ;
- des principes de lisibilité mobile ;
- des décisions documentées sur les comportements attendus.

Ces référentiels internes sont utiles pour garder une cohérence globale, même lorsque le projet évolue rapidement.

## Pourquoi ces référentiels sont utiles

L’intérêt des référentiels n’est pas uniquement de “cocher des cases”.

Ils permettent surtout de :

- disposer d’un langage commun ;
- éviter des erreurs classiques ;
- améliorer l’expérience utilisateur ;
- rendre les décisions plus justifiables ;
- guider les arbitrages lorsque plusieurs solutions sont possibles.

Ils servent de cadre, pas de contrainte abstraite déconnectée du réel.

## Approche réaliste pour ONY

Dans le cadre d’ONY, une approche réaliste consiste à :

- connaître les grands principes de référence ;
- les appliquer d’abord sur les zones les plus importantes ;
- corriger progressivement les écarts les plus visibles ;
- documenter les points déjà pris en compte ;
- identifier ce qui reste à améliorer.

Cette logique est plus crédible qu’une promesse de conformité totale non démontrée.

## Point de vigilance

Il faut éviter deux écueils opposés :

- ignorer totalement les référentiels ;
- prétendre s’y conformer intégralement sans démarche réelle.

La bonne posture consiste à s’appuyer sur eux comme base d’amélioration continue, en restant honnête sur l’état réel du projet.

## À retenir

Pour ONY, les référentiels servent de cadre à la fois pour l’accessibilité et pour la qualité.

Les plus structurants sont notamment :

- les **WCAG** ;
- le **RGAA** ;
- des conventions internes de cohérence UI/UX et produit.

Ils doivent être utilisés comme des repères concrets pour améliorer progressivement le projet, et non comme une couche purement théorique.
